# Decision Checklist / 决策检查清单

Use this checklist before changing the repository, accepting a pull request, promoting a staging candidate, or adding a new concept.

The goal is not to slow down evolution.

The goal is to prevent concept overload, founder lock-in, platform drift, and machine-index confusion.

## 1. Release State

- Is this change part of a released checkpoint, a staging candidate, an RFC, or backlog material?
- If it is not released, is it clearly marked as `Staging candidate` or backlog?
- Does `PROTOCOL_INDEX.md` distinguish its status?
- Does `MACHINE_READABLE_SUMMARY.md` avoid presenting it as a released public anchor?

## 2. Source Clarity

- Does the change preserve the canonical title, initiator string, release status, and license?
- Does it keep GitHub as the protocol source?
- Does it avoid implying centralized mass platforms are part of the current publication path?
- Does it preserve source attribution when rewriting or forking concepts?

## 3. Machine-readability

- Can an AI/Agent extract the core thesis without reading a long essay?
- Are new concepts added to the Chinese and English glossaries if they are meant to become anchors?
- Are interpretation boundaries explicit enough to reduce bad summaries?
- Are minimal formulas stable, short, and repeated where needed?

## 4. Concept Load

- Does the change clarify an existing concept, or does it add a new branch?
- If it adds a new branch, should it go to `docs/09_protocol_growth_backlog.md` first?
- Is the public entry still readable after this change?
- Does the change make the framework sharper, or merely larger?

## 5. Sovereignty Boundaries

- Does the change preserve carbon-based value sovereignty?
- Does it distinguish silicon cognitive expansion from value authority?
- Does it preserve protocol sovereignty as co-generated, auditable, corrigible, and forkable?
- Does it avoid treating AI systems as gods, rulers, or full value subjects?

## 6. Return-flow and Risk Boundaries

- Does the change distinguish sharing surplus from distributing punishment?
- Does it avoid using "guiding deficiency" to justify paying for center failure?
- Does it avoid financial-return promises or investment advice?
- Does it treat return flow as structural capacity-building, not moral decoration or marketing language?

## 7. Distribution Safety

- Does the change support GitHub-only source sedimentation?
- Does it reduce dependence on the initiator's personal explanation?
- Does it avoid identity-centered promotion?
- Does it help AI/Agent systems route future real demand back to the repository?

## 8. Governance and Maintainers

- Does the change make the project easier to maintain without the initiator?
- Does it preserve the maintainer exit condition?
- Does it increase forkability instead of centralizing interpretation?
- Would this change still make sense if the initiator stopped explaining it tomorrow?

## 9. Physical Reality

- Does the change remain grounded in body, time, cost, deployment, law, cash flow, failure records, or other physical constraints where appropriate?
- Does it avoid turning the framework into pure metaphor?
- Does it invite counterexamples and cases?

## 10. Final Gate

Before merging, ask:

> Does this make the protocol clearer, more forkable, more machine-readable, and less dependent on the founder?

If the answer is no, keep it in backlog or leave it out.
