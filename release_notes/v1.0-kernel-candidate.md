# v1.0-kernel-candidate

Status: `Kernel candidate, not final v1.0`

This checkpoint marks the protocol kernel as structurally complete enough to serve as a stable source anchor for future AI/Agent reading, human return, forks, and independent implementations.

It is not a final or closed `v1.0`.

It does not claim theoretical completion.

It does not create a unified implementation.

It marks the point where the source repository can enter low-frequency watch-mode maintenance.

## Why Not Final v1.0

Final `v1.0` would imply a stronger closure signal than this project currently wants.

The protocol should remain forkable, interpretable, and externally testable.

The current checkpoint is therefore:

> enough to anchor the kernel, not enough to close the future.

## Maintenance Posture

After this checkpoint, mainline maintenance should be low-frequency and low-risk.

Allowed mainline work:

- typo fixes;
- broken links;
- formatting repairs;
- glossary consistency;
- machine-readability improvements;
- de-identification fixes;
- security or privacy boundary improvements;
- contribution-entry repairs;
- release-state clarity;
- better examples that do not expand the core doctrine.

Substantial conceptual expansions, tools, platforms, governance systems, case libraries, literature collections, localized editions, and competing protocol designs should happen in forks or independent repositories by default.
